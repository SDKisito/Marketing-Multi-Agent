# Carrousel LinkedIn — Claude Code : CLAUDE.md, Skills & MCP expliqués

## Contexte
Carrousel éducatif LinkedIn (format 1:1) à destination de dev, tech leads et fondateurs tech curieux de Claude Code. L'objectif : décomposer les 3 piliers fondamentaux (CLAUDE.md, Skills, MCP) pour passer du « vibe coding » à un usage structuré et puissant.

## Framework utilisé
**PAS — Problem / Agitate / Solution**
La majorité des utilisateurs de Claude Code l'exploitent comme un autocomplete glorifié. On agite ce manque à gagner (les 90 % inutilisés), puis on offre la structure comme solution : les 3 piliers que tout dev devrait maîtriser.

## Hook — variante A *(la plus forte)*
> Vous utilisez Claude Code comme un copilote. En réalité c'est un *orchestrateur*.

## Hook — variante B
> 3 fichiers. Toute la puissance de Claude Code. La plupart des devs n'en connaissent qu'un.

## Deck carrousel

```markdown
# Vous utilisez 10 % de ce que Claude Code peut *faire*.
Voici les 3 piliers qui débloquent les 90 % restants — sans tout réécrire.

## [stat] Le vrai problème
90 %
Des devs utilisent Claude Code comme un autocomplete
Zéro mémoire projet. Zéro contexte partagé. Zéro automatisation. Résultat : on réexplique tout à chaque session.

## [toc] Les 3 piliers de Claude Code
1. CLAUDE.md — La mémoire permanente du projet
2. Skills — Les compétences spécialisées à la demande
3. MCP — La connexion aux outils externes

## _CLAUDE.md_ — La mémoire de votre projet
Un fichier markdown à la racine. Claude le lit à chaque démarrage.
Conventions de code, stack technique, commandes de build, règles d'architecture. Plus jamais besoin de réexpliquer.

## [flow] Ce que contient un bon CLAUDE.md
R · G · C — trois blocs, toujours.

1. Role — Le contexte du projet (stack, équipe, contraintes)
2. Guide — Standards de code, nommage, patterns à respecter
3. Commands — Les commandes clés : dev, build, lint, test

## [compare] CLAUDE.md *basique* vs structuré
Même projet. Résultats radicalement différents.

LEFT: Basique
- « Utilise React »
- « On préfère TypeScript »
- Claude oublie à chaque session

RIGHT: Structuré
- Stack complète + conventions documentées
- Patterns métier explicites (ex. toujours filtrer par date)
- Mémoire hiérarchique : global → projet → dossier

## Skills — des *compétences* que Claude active seul
Les Skills sont des fichiers markdown placés dans `.claude/skills/`.
Claude les charge à la demande selon le contexte. Pas besoin de les appeler manuellement.

## [pillars] Skills vs MCP vs Subagents
### Skills
Markdown de savoir-faire. Claude l'active si la tâche correspond. Portable entre projets.

### MCP
Connecteur vers des outils externes (GitHub, BDD, APIs). Fournit des données et des actions.

### Subagents
Agents isolés pour les tâches parallèles, avec leurs propres permissions d'outils.

## [valuechain] Les Skills se nourrissent de CLAUDE.md
HUB: CLAUDE.md | Règles permanentes du projet
- Skill Code Review | Hérite des conventions de code
- Skill Refactor | Connaît la stack et les patterns
- Skill Tests | Respecte les frameworks en place

## MCP — le *connecteur* universel vers vos outils
Model Context Protocol = adaptateur USB‑C pour l'IA.
Connectez GitHub, Playwright, vos APIs, vos bases de données. Claude accède à tout via des commandes slash.

## [matrix] Quel outil pour quel besoin
X: Usage | ponctuel | permanent
Y: Nature | données externes | savoir-faire interne
TL: MCP (GitHub, BDD, APIs)
TR: Skills permanents
BL: Requêtes manuelles
BR: CLAUDE.md

## [compare] Sans vs *avec* les 3 piliers
Même équipe. Productivité incomparable.

LEFT: Sans
- Réexpliquer le contexte à chaque session
- Zéro cohérence entre devs
- Claude ignore vos outils internes

RIGHT: Avec
- Mémoire projet partagée (CLAUDE.md versionné)
- Skills réutilisables entre tous les agents
- MCP connecte vos outils en quelques lignes

## [thanks] *Maîtriser* Claude Code — par où commencer ?
Créez votre CLAUDE.md aujourd'hui. Documentez votre stack. Ajoutez un premier Skill.
- naiomagency.com
- Save · Share · Commentez votre setup 👇
